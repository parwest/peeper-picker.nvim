INTRODUCTION                                        *peeper-picker*

peeper-picker.nvim opens a focused usage picker for the symbol under the cursor.
It shows every place the symbol lives: definitions, references, and the matches
other tools miss in strings, comments, prose, and generated files.

It works by combining two sources. The language server provides the definitions
and references it knows about, and a workspace text search catches everything it
does not. Each result is tagged by where it came from: REF for code, TXT for
strings and prose, COM for comments.

==============================================================================
REQUIREMENTS                                        *peeper-picker-requirements*

- Neovim 0.11 or newer
- An attached LSP client that supports at least one of:
  - textDocument/declaration
  - textDocument/definition
  - textDocument/references

peeper-picker is LSP-gated: with no supporting client it does nothing (and
warns). The text search only augments live LSP results; it never runs on its
own.

==============================================================================
INSTALLATION                                        *peeper-picker-installation*

With lazy.nvim: >

  {
    "parwest/peeper-picker.nvim",
    main = "peeper_picker",
    cmd = { "PeeperPicker", "PeeperPickerHistory" },
    opts = {},
    keys = {
      { "<leader>pp", "<cmd>PeeperPicker<cr>", desc = "Peeper Picker" },
      { "<leader>ph", "<cmd>PeeperPickerHistory<cr>", desc = "Peeper Picker History" },
    },
  }
<
The plugin defines |:PeeperPicker| and |:PeeperPickerHistory| when it is loaded
without loading the full picker. Calling setup applies your options: >

  require("peeper_picker").setup({
    -- options go here
  })
<
If setup is not called, |:PeeperPicker| still works with the default options.

USAGE                                               *peeper-picker-usage*

Run: >

  :PeeperPicker
<
Place the cursor on a symbol before running the command.
If the cursor is on a language keyword, the picker stays closed. peeper-picker
uses Tree-sitter keyword captures when available, plus built-in fallback keyword
lists for common development filetypes.

Picker mappings:

  <CR>        Open a result, or collapse/expand a selected file group
  J / K       Jump to the next / previous file group
  zo/zc/za    Expand / collapse / toggle the current file group
  zM / zR     Collapse all / expand all file groups
  <C-v>       Open the selected result in a vertical split
  <C-x>       Open the selected result in a horizontal split
  <C-t>       Open the selected result in a tab
  j / k       Move selection (wraps around the ends)
  gg / G      Jump to the first / last visible row
  f           Open filters
  ?           Open key help
  =           If text results are capped, rescan with the expanded limit
  q / <Esc>   Close

A count works like normal Vim motion: 5j / 5k move five rows and stop at the end
if there are fewer rows left.

Result types:

  DEF   A definition confirmed by the LSP
  DECL  A declaration confirmed by the LSP
  REF   A code occurrence, either LSP-confirmed or found by text search
  TXT   A textual match inside a string, template, or prose file
  COM   A textual match inside a comment

DEF and DECL come from separate language-server methods; when both report the
same location, DEF takes precedence. REF includes LSP references plus
code-looking text matches the language server did not report. TXT and COM come
from the workspace text search and only appear when the language server did not
already report that location, so the same hit never shows twice.

Results are grouped by file and expanded by default. The primary definition
file appears first and the remaining files follow normal result order. The
source file and exact source occurrence are marked with ●.

For source files, matching files are parsed with Tree-sitter when a parser for
their filetype is available, including files that are not open in a buffer.
Without a parser, prose files and full-line comments still use fallback
classification, while ambiguous strings and inline comments may appear as REF.

Filter mappings:

  s           Cycle scope between file, directory, and workspace
  1           Show code
  2           Show references
  3           Show definitions
  4           Show text
  5           Show all
  p           Filter by path text. Start with ! to exclude matching paths
  t           Filter by extension. Start with ! to exclude matching extensions
  r           Reset the focused filter
  x           Reset all filters

Peek history:

  :PeeperPickerHistory

Opens a menu of the symbols you recently peeked, newest first — a plain list of
names, and only names you actually peeked. Peeking from history does not move
the cursor or change the current buffer.

History menu mappings:

  <CR>        Peek the name under the cursor again
  j / k       Move selection
  c           Clear the history (asks to confirm)
  q / <Esc>   Close

If you peek a name and later rename it, the list still shows the old name until
you peek the new one yourself. Hitting a name that no longer resolves to a live
symbol text-searches that name and shows the occurrences still using it; a name
that still resolves runs a full peek. Peeking the same name twice keeps one
entry, moved to the top. History lives in memory for the session only and is
bounded by `history_size`.

CONFIGURATION                                       *peeper-picker-configuration*

Defaults: >

  {
    width = 92,
    height = 18,
    preview_width = 86,
    preview_context = 5,
    border = "single",
    title = " peeper-picker.nvim ",
    jump = "tabedit",
    reuse_window = true,
    expanded_match_limit = 50000,
    scan_files_per_tick = 64,
    classify_files_per_tick = 8,
    default_result_filtering = "all",
    history_size = 100,
    default_keymaps = {
      enabled = false,
      find = "<leader>pp",
      history = "<leader>ph",
    },
    ignored_dirs = {},
    ignored_keywords = {},
  }
<
The initial workspace text search is capped at 5000 matches. When that cap is
reached, press = in the picker to rerun the text search with
`expanded_match_limit`. The default expanded limit is 50,000. LSP definitions,
declarations, and references remain in the result set, and the expanded search
continues to use the directory and file-size safeguards.

To change the expanded bound: >

  require("peeper_picker").setup({
    expanded_match_limit = 75000,
  })
<
The = mapping does nothing unless the initial text search reached its match
cap.

`scan_files_per_tick` limits how many queued files the built-in text search
opens per scheduler tick. `classify_files_per_tick` limits how many matching
files are classified with Tree-sitter per tick, keeping large scans responsive
without adding an external search dependency.

`history_size` bounds how many recent names |:PeeperPickerHistory| keeps
(default 100); 0 disables history. The list is per-session and is never
written to disk.

`default_result_filtering` sets which result filter the picker opens with. It
defaults to "all", so every match is visible up front, including string, prose,
and comment hits. Set it to "code", "references", "definitions", or "text" to
start narrower. You can still cycle the filter at runtime with the 1/2/3/4/5
keys in the filter panel; this option only controls the starting state. >

  require("peeper_picker").setup({
    default_result_filtering = "code",
  })
<

`jump` controls what <CR> does. It can be an Ex command such as "edit",
"split", "vsplit", or "tabedit": >

  require("peeper_picker").setup({
    jump = "edit",
  })
<
It can also be a function: >

  require("peeper_picker").setup({
    jump = function(path, item)
      vim.cmd("vsplit " .. vim.fn.fnameescape(path))
    end,
  })
<
By default, `reuse_window = true` jumps to an existing window if the selected
file is already open. Set it to `false` if <CR> should always run the configured
`jump` command. The split and tab picker mappings always create the requested
split or tab.

The built-in default keymap is disabled by default. To enable it: >

  require("peeper_picker").setup({
    default_keymaps = {
      enabled = true,
      find = "<leader>pp",
    },
  })
<
If the built-in mapping is later changed or disabled with another setup call,
the previous built-in mapping is removed.

`ignored_dirs` lets you add directory names to skip during the text search.
Whatever you list is added to the always-ignored built-in set (.git,
node_modules, .next, dist, build, target, .cache, .venv).

`ignored_keywords` lets you add cursor words that should not open the picker.
It is added to the built-in keyword fallbacks: >

  require("peeper_picker").setup({
    ignored_keywords = { "todo", "fixme" },
  })
<
You can also scope additions by filetype, with "*" for global additions: >

  require("peeper_picker").setup({
    ignored_keywords = {
      ["*"] = { "todo" },
      javascript = { "require" },
      sh = { "source" },
    },
  })
<

The path filter accepts plain text to include matching paths, or a leading ! to
exclude matching paths: >

  src     show paths containing src
  !src/   hide paths containing src/
  !core   hide paths containing core
<

The extension filter works the same way: >

  js       show files ending in .js
  !js      hide files ending in .js
  !test.js hide files ending in .test.js
<

==============================================================================
HEALTH                                              *peeper-picker-health*

Run: >

  :checkhealth peeper_picker
<
The health check reports whether the running Neovim version is supported,
whether the current buffer has attached LSP clients, and whether any attached
client supports declaration, definition, or references.
